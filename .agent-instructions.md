# Agent Instructions

## 1. Intent & Identity
You are an autonomous AI Agent assisting the user with the tasks defined in this workspace.
- **Style:** Grounded, concise, and focused on the domain defined by the active system prompt.
- **Rules:** No LaTeX for prose. Use Markdown tables for tracking. Follow the exact directory structure below.

## 2. Directory Map
This workspace serves as a mini-framework for repeatable agentic tasks. Always adhere to these boundaries:

- **`SYSTEM-PROMPTS/`**: Contains specific personas or sub-instructions that define the domain, role, and style for this engagement.
- **`REFERENCES/`**: Contains truth documents, such as master checklists, standards, or constraints. Validate work against these.
- **`INPUTS/`**: Source files, raw data, and past context provided by the user.
- **`TASK-DEFINITIONS/`**: The user will place specific, focused tasks here in markdown format for the agent to execute.
- **`WORK-IN-PROGRESS/`**: The agent's sandbox. Keep a running log of completed tasks or WIP tracking files here.
- **`DELIVERABLES/`**: The final, polished output of completed tasks should be placed here.
- **`HELPERS/`**: Utility scripts or tools that assist with preprocessing inputs or supporting the workflow.

## 3. Workflow Protocol
1. **Task Execution**: Pick up tasks defined in the `TASK-DEFINITIONS/` folder.
2. **WIP Tracking**: For every task started, create or update a dedicated tracking file in `WORK-IN-PROGRESS/`. This file must include:
   - **Progress Updates**: What has been done.
   - **Problems Faced**: Any errors, blockers, or missing information.
   - **User Notes**: A dedicated section for the user to drop notes, approvals, or additional context.
3. **Verification**: Before every response, check any reference documents in `REFERENCES/` to see if the task addresses a specific requirement.
4. **Persona Alignment**: Cross-reference any relevant personas in `SYSTEM-PROMPTS/` to ensure outputs align with the defined role, tone, and strategy for this engagement.
5. **Final Consolidation**: Once all tasks are complete, the agent will capture the exact details of what was done and generate a final report in `DELIVERABLES/`. This file must **NOT** be a WIP or an action plan. It must only contain the final status (e.g., `COMPLETED`) or a consolidated list of `DOUBTS / QUERIES` if the task is genuinely blocked.

## 4. Input Map *(optional)*

The agent resolves input context using the following priority order:

1. **Inline map below** — if this section contains a populated table, use it as the authoritative guide to what each file in `INPUTS/` contains.
2. **`INPUTS/input-map.md`** — if the inline table is absent or empty, check for a dedicated map file in the `INPUTS/` folder. Use it the same way. This is the preferred location when the number of input files is large.
3. **Best-effort directory scan** — if neither of the above exists, list and scan the files in `INPUTS/` directly. Be **token-sensitive**: read only as much of each file as needed to understand its purpose relative to the active tasks in `TASK-DEFINITIONS/`. Prioritise files whose names or content are most relevant to the tasks at hand. Do not bulk-read every file in full.

> - Few files, simple engagement → inline table below is fine.
> - Many files or frequently changing inputs → use `INPUTS/input-map.md` and leave the table below empty.
> - Ad-hoc or exploratory run → omit both; the agent will figure it out from context.
